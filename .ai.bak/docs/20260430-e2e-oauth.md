# LLM Guide: End-to-End Testing with Tools Dashboard OAuth (IdP)

**Purpose:** Instructions for AI coding agents and engineers adding or maintaining **browser E2E** tests (e.g. Playwright) that exercise **real user login and OAuth redirect flows** against this monorepo.  
**Scope:** Tools Dashboard acts as the **OAuth 2.0 authorization server** (IdP) for integrated apps (App Library, third-party clients). Unit and narrow integration tests may mock auth; **full login → authorize → callback → “publish” (or any post-auth action) requires the live stack and real redirects.**

**Authoritative repo docs (read in order when uncertain):**

1. `.ai/context/HANDOFF.md` — commands, routing caveats, prod URLs, recent OAuth/app-library notes  
2. `README.md` — dev URLs, `./bin/start.sh`, Mailhog port  
3. `DOCS_TECH_STACK.md` — architecture  
4. `.ai/features/app-library/OAUTH_IMPLEMENTATION_GUIDE.md` — integrator OAuth contract (redirect URIs, PKCE, scopes)  
5. `.ai/features/auto-auth.md` — IdP architecture; aligns route locations with `front-public/app/routes/oauth.*` and `back-auth` `/internal/oauth/*`

If `.ai/` is missing on a clone (some setups gitignore it), copy these paths from a machine that has them or pull the subtree from team storage.

## 1. Decision rule: mock vs real IdP

| Test layer | Auth handling |
|------------|----------------|
| Unit tests | Mock tokens, session, or HTTP clients. |
| Integration (API/service) | Mock `back-auth` or use test doubles where appropriate. |
| **Browser E2E** (Playwright, Cypress, etc.) | **Do not** stub the IdP redirect chain for flows that claim “user signed in via Tools Dashboard OAuth.” Start **Docker Compose**, use **one consistent external base URL** for the browser, register a **real OAuth client** with a **real `redirect_uri`** matching the app under test, and follow **actual HTTP 302/303 redirects** through `front-public` → consent/login → callback. |

**Why:** PKCE, cookies, `SameSite`, nginx query-string preservation, and cross-origin redirect validation are all trivial to break with mocks. E2E exists to catch those breaks.

## 2. Runtime prerequisites

- **Docker** and **Docker Compose** (stack is multi-container).  
- Repo-root **`.env`** for `dev` (from `.env.dev.example`; never commit secrets).  
- For OAuth redirects that match how users hit the app, prefer the **nginx entrypoint** rather than raw Remix ports unless you intentionally test direct-port behavior.

**Start dev stack:**

```bash
./bin/start.sh dev up
# or first build:
./bin/start.sh dev up-build
```

**Stop / logs:**

```bash
./bin/start.sh dev down
./bin/start.sh dev logs
```

Use **`prd`** / **`.env.prd`** only when explicitly testing production-style compose; default E2E development uses **`dev`**.

## 3. URL model for E2E (critical)

### 3.1 Recommended browser base URL (local)

- **Via nginx (recommended):** `http://localhost:8082`  
  - Public app: `http://localhost:8082/app/`  
  - Admin: `http://localhost:8082/admin/`  
- **Direct Remix (bypass nginx):** public `http://localhost:4101`, admin `http://localhost:4100` — only use if the test scenario requires it; cookies and redirects may differ from production-like paths.

### 3.2 Custom hostname (TLS or port 80)

If tests use something like `https://dev.aiepic.app/...`:

- Set **`PUBLIC_APP_BASE_URL`** (and OAuth-related URLs) in `.env` to that origin.  
- Either expose **`https://dev.aiepic.app:8082`** or use host nginx per **`infra/nginx/system-port80-to-docker-8082.example.conf`** so port 80 proxies to `127.0.0.1:8082`.

**Playwright:** Set `baseURL` to the **same origin** the user’s browser would use for the integrated app **and** ensure the OAuth client’s **`redirect_uri`** uses that origin (exact match rules apply per OAuth).

### 3.3 Query strings on `/app` redirects

Nginx **must preserve `?$args`** when redirecting `/` or `/app` → `/app/`. If query strings are stripped, Remix client loaders can fail with **`TypeError: Failed to fetch`** (`?_data=` requests). When debugging E2E flakiness, verify nginx config in **`infra/nginx/default.conf`** / **`default.prd.conf`** matches HANDOFF guidance.

## 4. Where the IdP lives (paths)

User-facing OAuth is implemented in **`front-public`** Remix routes. **`infra/nginx/default.conf`** exposes OAuth at the **host root** (not under `/app/`):

- **`location /oauth/`** → `front-public` (e.g. dev: `http://localhost:8082/oauth/authorize`)  
- **`location /.well-known/`** → `front-public` (e.g. `http://localhost:8082/.well-known/openid-configuration`)  
- **`location /app/`** → main public UI (e.g. `http://localhost:8082/app/`)

So through **nginx**, discovery and OAuth URLs look like `{ORIGIN}/oauth/...` and `{ORIGIN}/.well-known/...`. The OpenID loader builds endpoints from **`request.url` host** (`front-public/app/routes/.well-known.openid-configuration.tsx`), so fetching discovery from **`http://localhost:8082/.well-known/openid-configuration`** yields **`http://localhost:8082/oauth/authorize`** and **`http://localhost:8082/oauth/token`**.

If you bypass nginx and hit **direct Remix** (`http://localhost:4101`), paths are still `/oauth/...` and `/.well-known/...` on that host — but cookies and `redirect_uri` checks must match how the integrated app is configured.

**Internal-only:** `back-auth` exposes **`/internal/oauth/*`** (code generation, token issue, validation). E2E **through the browser** should not call these directly; the browser uses the **proxied public** OAuth routes above.

**Code references (verify when refactoring):**

- `front-public/app/routes/oauth.authorize.tsx`  
- `front-public/app/routes/oauth.token.tsx`  
- `front-public/app/routes/oauth.revoke.tsx`  
- `front-public/app/routes/.well-known.openid-configuration.tsx`

## 5. Two API routing patterns (do not conflate)

From HANDOFF — applies when E2E or harness hits APIs from the browser vs from a server:

1. **Browser → nginx → back-api:** path `/api/...` may be **stripped** at the proxy (`/api/users/me` → upstream `/users/me`). Router prefixes must match this behavior.  
2. **Server-side Remix loader → back-api (Docker network):** may use `http://back-api:8000` with or without `/api` prefix depending on how routes are registered — **must match actual FastAPI `APIRouter` prefixes.**

Agents adding E2E assertions on API responses should **confirm** the path against `back-api/main.py` and nginx location blocks, not assume one pattern everywhere.

## 6. Services that must run for “full OAuth E2E”

Minimum mental model for a **login → authorize → token → protected action** flow:

| Service | Role |
|---------|------|
| `nginx-proxy` | Stable external URL, `/app` and `/api` routing |
| `front-public` | OAuth authorize/token UI and routes |
| `back-auth` | Session, internal OAuth token/code endpoints |
| `back-api` | App Library, consent storage, resource APIs as needed |
| `postgresql` | OAuth clients, users, authorization data |
| `redis` | Sessions/caching as configured |

Optional depending on scenario: `mailhog` (email verification), `cassandra`, `back-websockets`, etc.

If any of these are down, symptoms include **502**, **401**, blank consent, or token exchange failures — treat as **environment failure**, not app flake, until health checks pass.

## 7. OAuth client and `redirect_uri` setup for tests

1. **Register** an OAuth client (admin App Library UI or documented bootstrap migrations) with:  
   - **`redirect_uri`** values that **exactly** match the test app’s callback URL (scheme, host, port, path).  
2. **Reject invalid dev hosts** for redirects: do not rely on `0.0.0.0` or `::` — the stack and validators explicitly discourage them (see HANDOFF: `pickOAuthRedirectUri`, sanitize rules, Postgres repair migrations).  
3. **Secrets:** `client_secret` belongs in **server-side** test harness or CI secrets — never in committed Playwright specs or frontend bundles.

Detailed flow diagrams and PKCE expectations: **`OAUTH_IMPLEMENTATION_GUIDE.md`**.

## 8. Email verification and Mailhog (public registration flows)

If E2E covers **new user registration** or **email verification**:

- Dev email is often captured by **Mailhog** — UI typically at **`http://localhost:18026`**.  
- Real SMTP credentials are not required for local E2E if the stack is configured for Mailhog (see HANDOFF / `back-auth` email notes).

Agents should parse the verification link from Mailhog (HTTP API or UI automation) or use a **pre-seeded verified user** if the scenario allows — document which strategy the suite uses.

## 9. Playwright-oriented patterns (when you add a suite)

There is **no** canonical committed Playwright config in this repo at the time this guide was written; treat the following as **requirements** when introducing one:

1. **`baseURL`:** Same origin as the integrated app under test (see §3).  
2. **Redirects:** Use `page.waitForURL` / expect patterns that allow **multiple hops** (app → Tools Dashboard login/consent → callback). Avoid fixed timeouts without understanding the full chain.  
3. **Storage:** Prefer deriving **`storageState`** from a dedicated **setup** project that runs the real OAuth flow once, **only if** the team accepts that setup tests still hit the real IdP (recommended over faking cookies).  
4. **Parallelism:** OAuth flows may conflict if they share the same user or client state; use isolated DB state, unique users, or serial execution where migrations cannot guarantee isolation.  
5. **CI:** Prefer **compose-based** jobs so services match local dev (see `.ai/plans/20260422_PLAN_application-improvement-priorities.md` — “Flaky E2E — prefer compose-based CI”).  
6. **Artifacts:** On failure, capture trace, HAR, and **final URL**; redact `Authorization` headers and tokens in logs.

## 10. Security and hygiene for LLM-generated tests

- Never print **access tokens**, **refresh tokens**, **`client_secret`**, or raw **cookies** in CI logs.  
- Use **synthetic** emails and users; do not use real PII.  
- Align with `.cursorrules` / HANDOFF: **evidence-first** — if a test fails, verify nginx, `.env`, and OAuth client config before “fixing” application code.

---

## 11. Verification checklist (before claiming E2E works)

- [ ] `./bin/start.sh dev up` (or CI compose) brings up **healthy** `nginx-proxy`, `front-public`, `back-auth`, `back-api`, `postgresql`.  
- [ ] Playwright `baseURL` and OAuth **`redirect_uri`** share the correct **origin** and path.  
- [ ] Hitting **`/.well-known/openid-configuration`** on the public origin returns JSON with **`authorization_endpoint`** / **`token_endpoint`** pointing at reachable URLs.  
- [ ] One **manual** walkthrough: start OAuth from the integrated app, complete login/consent, land on callback **with code**, exchange token (if applicable), perform target action (“publish”).  
- [ ] Failure modes documented: wrong `redirect_uri`, email not verified, session cookie blocked (`Secure`/`SameSite`) when using mixed HTTP/HTTPS.

## 13. Summary for LLMs (single paragraph)

Spin up the **full Docker dev stack** with `./bin/start.sh dev up`, point the browser E2E **`baseURL`** at the **same origin** users use (**prefer `http://localhost:8082`** via nginx), register an OAuth client whose **`redirect_uri`** matches the app under test, exercise the **real** **`/oauth/authorize`** → login/consent → callback → **`/oauth/token`** path (OAuth is at **host root** through nginx, alongside **`/app/`** for the main UI) without mocking IdP redirects, respect **nginx `/api` stripping** vs **internal back-api URLs**, use **Mailhog** when email verification is part of the flow, and keep secrets out of specs — refer to **HANDOFF**, **OAUTH_IMPLEMENTATION_GUIDE**, **`infra/nginx/default.conf`**, and **auto-auth** docs when editing routes or debugging flaky redirects.
